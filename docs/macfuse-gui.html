macFUSE GUI for macOS

A macFUSE GUI is the layer that makes SSHFS mounts practical on macOS without living in the terminal. macFUSEGui sits on top of macFUSE and sshfs, then gives you a menu bar workflow for connecting, disconnecting, monitoring, and recovering mounts.

Download Latest Release View Source Open Install Guide

macFUSE

macFUSE provides the filesystem layer that lets remote content appear as a normal folder on macOS.

sshfs

sshfs is the transport that mounts an SSH-accessible path into Finder and your editor using FUSE.

macFUSEGui

macFUSEGui is the operational layer: it stores credentials safely, manages remotes, and handles reconnect and diagnostics.

Why use a GUI instead of raw sshfs commands?

A shell-only workflow is fine when you mount one server once in a while. It gets brittle when you manage several remotes, switch networks, close laptops, or need quick visibility into whether a mount is healthy.

  • Per-remote connect and disconnect actions instead of rebuilding mount commands by hand.
  • Keychain-backed password storage so secrets do not sit in copied command history.
  • Automatic recovery after sleep, wake, network restoration, or unexpected unmounts.
  • Diagnostics output you can copy when an SSHFS mount fails or hangs.
  • Separate Apple Silicon and Intel builds so you install the correct binary for your Mac.

Install prerequisites

macFUSEGui does not replace macFUSE or sshfs. It manages them. Install both prerequisites first, then choose the correct app build for your Mac.

brew install --cask macFUSE
brew install gromgit/fuse/sshfs-mac
uname -m

If uname -m prints arm64, use the Apple Silicon build. If it prints x86_64, use the Intel build.

First launch on macOS

Current public builds are unsigned and not notarized. On first launch, open the app from Finder with right-click or Control-click and choose Open. If macOS blocks the launch, approve it from System Settings > Privacy & Security.

Once that one-time approval is complete, you can add a remote, test the connection, and mount it like a normal folder.

Typical daily workflow

  1. Save a remote with hostname, username, auth mode, and local mount point.
  2. Test the connection from the app before you rely on the mount in Finder or an editor.
  3. Connect when you start work, open the mounted path in Finder or your editor, and disconnect when finished.
  4. Let the app handle wake, sleep, and network recovery for desired remotes instead of remounting everything manually.

When to use the troubleshooting guide

Jump to troubleshooting if mounts fail after first launch approval, credentials are correct but mounts still do not appear, or a previously healthy remote stops responding after sleep or a network change.
